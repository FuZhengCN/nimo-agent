## 一、架构设计

### 亮点 ✅

**依赖链单向无循环** — `main → agent → llm/client + memory + tools → engine` 清晰分层，所有依赖关系在 CLAUDE.md 中有文档化的依赖链图。

**两层工具架构（透传→意图）** — 从 `tapd_cli`/`svn` 透传模式到 `tapd_query`/`svn_op` 意图模式的迁移方向正确。LLM 通过结构化 Intent 表达"做什么"，ExecutionEngine 确定性代码负责"怎么做"，LLM 不再直接操控 CLI 参数。

**编排与执行分离** — `ExecutionEngine` 单一职责，接收 `Intent` 数据类，按 `tool` 字段分发到 `_execute_tapd`/`_execute_svn`。内置 `for_each_workspace` 可复用执行模式，部分成功算成功。

**Skill 系统渐进式披露** — L1 元数据始终在 system prompt（~100字/Skill），L2 完整指令激活后注入，L3 脚本输出按需获取。三级降级解析（skill.yml → SKILL.md frontmatter → 目录名兜底）保证永不加载失败。

**工具自动发现** — `tools/__init__.py` 通过 `pkgutil.iter_modules` 自动加载，新增工具只需在 `nimo/tools/` 下创建模块 + `@register_tool` 装饰器，不改任何中央配置。

### 问题 ❌

1. **`main.py` 过于臃肿（433行，全项目最大文件）**
   - 混合了输入循环、内置命令处理（/help /chain /clear /exit）、Spinner 动画、chain 格式化（`_format_chain` + `_format_tool_call`）、定时任务通知展示（`_show_notification`）、Skill 安装/卸载/列表、`_input_with_poll` 轮询逻辑等多项职责
   - 建议拆分：`cli/commands.py`（内置命令）、`cli/spinner.py`（动画）、`cli/chain.py`（格式化）、`cli/input.py`（轮询输入）

2. **`engine.py` 中 `_build_tapd_args` 方法分支过多（386行模块）**
   - 每个 action 的参数组装逻辑不同（list/create/update/timesheet_add），当前用 if/elif 链处理，后续添加新 action 会更臃肿
   - 可考虑策略模式或 action 参数配置化

3. **模块级全局变量**
   - `_scheduler`（main.py:253）— 不利于测试隔离
   - `_config`（schedule.py:20）— 模块级可变状态
   - `_BIN_DIR`、`_TAPD_BIN`、`_SVN_EXE`、`_SVNADMIN_EXE`（engine.py:15-18）— 无法在测试中替换路径

4. **engine.py 中 `_run_tapd` 和 tapd.py 中 `tapd_cli` 存在代码重复**
   - 两者都做了：env 注入 TAPD_ACCESS_TOKEN → `asyncio.create_subprocess_exec` → communicate → decode → 错误处理
   - `engine.py` 的 `_run_tapd` 是意图工具路径，`tapd.py` 的 `tapd_cli` 是透传工具路径，逻辑高度相似但未共享

## 二、代码质量

### 亮点 ✅

- **类型注解覆盖率 ~88%** — 138 个函数中 121 个有返回类型注解
- **无通配符导入** — 所有 import 明确指定符号
- **无 TODO/FIXME 残留** — 代码干净，没有遗留标记
- **一致的错误处理模式** — 各模块统一使用 `ToolResult(success, data, error)` 三元组
- **一致的日志模式** — 所有模块使用 `logging.getLogger(__name__)`

### 问题 ❌

1. **23 处 `except Exception` 宽泛捕获**
   - agent.py: 5处（历史加载、档案加载、摘要生成、档案提取、LLM总结调用）
   - engine.py: 2处（`_run_tapd`、`_run_svn`）
   - registry.py: 2处（init_all、execute）
   - skill/registry.py: 7处（discover、activate、run_script 等）
   - main.py: 1处（主循环）
   - schedule.py: 2处（_execute、_check_all）
   - 建议细化为具体异常类型，避免吞掉意外错误

2. **未配置任何 Linter**
   - 项目没有安装 ruff、pylint、pyright 等代码质量工具
   - `pyproject.toml` 缺少 `[tool.ruff]`、`[tool.pyright]` 配置段

3. **部分文件缺少模块级 docstring**
   - agent.py、config.py、main.py、schedule.py 等核心模块没有文件头 docstring
   - 相比之下 engine.py、display.py、skill/registry.py 等有清晰的模块 docstring

4. **`main.py:425` 不一致的 logger 使用**
   - `logging.getLogger(__name__).exception("未预期的错误")` — 在 except 中重新调用 `getLogger` 而非使用模块级的 `logger`
   - 其余地方均使用模块级 `logger`，此处不一致

5. **`schedule.py` 文件头使用注释而非 docstring**
   - 第一行是 `# nimo/tools/schedule.py` 注释，而非模块 docstring

## 三、安全性

### 亮点 ✅

- **无硬编码密钥** — token/api_key 全部通过 config.yaml + `_env_override()` 环境变量覆盖机制注入
- **所有 subprocess 调用使用 `asyncio.create_subprocess_exec`** — 无 `shell=True`，参数以列表传递，无命令注入风险
- **路径遍历防护** — 三处均有校验：
  - `tapd.py:_validate_args()` — `os.path.normpath` 检测 `..`
  - `engine.py:_resolve_svn_path()` — 检测路径遍历模式
  - `skill/registry.py:run_script()` — 白名单 + 路径遍历校验
- **`.gitignore` 配置完善** — 排除了 config.yaml、.env、bin/*.exe、__pycache__、.venv 等

### 问题 ⚠️

1. **Token 泄露风险（已知未修复）**
   - `tapd.py:72-73` 注释标注：TAPD_ACCESS_TOKEN 通过环境变量传给外部二进制 `tapd.exe`
   - 若该二进制或其依赖库输出环境信息到 stdout/stderr，Token 存在泄露风险
   - `engine.py:_run_tapd` 同样存在此问题（相同的 env 注入模式）

2. **`python_exec` 工具无沙箱限制**
   - 允许 LLM 生成任意 Python 代码并通过 `sys.executable -c` 执行
   - 无网络访问限制、文件系统访问限制、进程创建限制
   - 这是设计意图，但风险需要明确认知

3. **`skill/registry.py:run_script()` 执行脚本时传入 `env={}`**
   - 空环境变量隔离了父进程环境，但也可能阻断脚本的正常运行（某些脚本可能依赖 PATH 等基础环境变量）

## 五、性能

### 亮点 ✅

- **工具并行执行** — `asyncio.gather` 并行执行多个 tool call，而非串行
- **滑动窗口 + 摘要压缩** — 历史消息超出限制时触发 LLM 摘要压缩，避免上下文爆炸
- **后台线程 Spinner** — `main.py:_Spinner` 在独立 daemon 线程运行，不阻塞 Agent 循环
- **120s 超时保护** — 所有 subprocess 调用和工具执行均有超时限制
- **原子文件写入** — `history.py:save()` 和 `schedule.py:_save_schedules()` 使用 .tmp → rename 模式防损坏

### 问题 ⚠️

1. **Scheduler 固定 60s 轮询间隔**
   - 对 `delay_minutes` 一次性延迟任务不精确，实际触发时间 = 期望时间 + 0~60s
   - 可考虑对 once 任务计算精确的 sleep 时间

2. **`python_exec` 无输出大小限制**
   - LLM 生成的代码可能产生大量输出，当前无截断机制
   - 对比：`_build_summary_prompt` 对 tool 消息做了 500 字符截断
